Lộ trình học Java Backend từ con số 0

Dành cho dev Angular/TypeScript · thực hành trực tiếp trên project BE/ (Spring Boot, salary-service)

Chào mừng bạn đến với "Lộ trình học Java Backend từ con số 0" — tài liệu được viết RIÊNG cho người như bạn: một lập trình viên frontend đã rành Angular 17 và TypeScript, nhưng chưa từng chạm tới Java hay backend. Tin tốt là: bạn không học từ con số 0 thật sự. Bạn đã sở hữu phần khó nhất — tư duy về component, dependency injection, interface, generic, kiểu dữ liệu tĩnh và cách gọi REST API. Java và Spring Boot chỉ là một "phương ngữ" mới để diễn đạt những ý tưởng bạn vốn đã hiểu. Trong suốt tài liệu, mỗi khái niệm Java mới sẽ được giới thiệu bằng cách neo vào thứ bạn đã quen ở Angular/TS: @Component ↔ @RestController, @Injectable + constructor injection ↔ @Service + constructor injection của Spring, interface/generic của TS ↔ interface/generic của Java, HttpClient ↔ RestTemplate/Feign, package.json + npm ↔ pom.xml + Maven. Mục tiêu là bạn học NHANH bằng cách "dịch" kiến thức cũ, thay vì học lại từ đầu.\n\nĐiểm khác biệt lớn nhất của tài liệu này so với các khóa Java thông thường: chúng ta KHÔNG học lý thuyết suông trên ví dụ đồ chơi. Toàn bộ lộ trình thực hành trên một project Spring Boot CÓ THẬT đang chạy production — **salary-service** (thư mục BE/), một dịch vụ quản lý lương và hoa hồng. Đây là project thật với Spring Boot 3.4.7, Java 17, Maven 3.9.10 (qua Maven Wrapper), cơ sở dữ liệu MariaDB, có đầy đủ các tầng Controller → Service → Repository → Entity, có Spring Security (theo mô hình Gateway truyền header), có xử lý lỗi tập trung (GlobalExceptionHandler), i18n 4 ngôn ngữ, và 57 entity JPA mô tả nghiệp vụ lương thực tế. Bạn sẽ đọc, chạy, sửa và mở rộng code thật — đúng như cách một backend dev đi làm thực thụ.\n\nLộ trình gồm 11 module, đi từ "chiến thắng đầu tiên" (cài đặt và chạy được app trên máy) cho tới khi bạn đủ tự tin tự xây một backend hoàn chỉnh. Bốn module đầu (0–3) xây nền: cài JDK/Maven, làm chủ cú pháp Java cho người quen TypeScript, nắm OOP và các idiom Java (class, interface, generic, collection), rồi hiểu trái tim của Spring Boot là Dependency Injection — thứ bạn đã dùng hàng ngày trong Angular. Các module 4–7 đi xuyên từng tầng của một REST API thật: Controller, tầng dữ liệu JPA & Spring Data, tầng Service & nghiệp vụ, và cách chuẩn hoá response, xử lý lỗi, i18n (đúng theo ExecutionResult + GlobalExceptionHandler của salary-service). Ba module cuối (8–10) đưa bạn tới các chủ đề "đi làm mới gặp": bảo mật cơ bản với Spring Security, cấu hình & hạ tầng (profile dev/prod, Actuator, bật/tắt tính năng như Redis/Kafka), kiểm thử, và khép lại bằng lộ trình thực chiến: tự xây dựng một backend mới cho chính ứng dụng học tiếng Anh (eng) mà bạn đang làm ở frontend.\n\nSau khi hoàn thành, bạn sẽ không chỉ "biết Java" — bạn sẽ đọc hiểu được một codebase Spring Boot thật, biết một HTTP request đi qua những tầng nào, tự thêm được một endpoint mới từ Controller xuống tận database, và quan trọng nhất: tự tin trở thành một fullstack developer thực thụ, kết nối được frontend Angular bạn đã giỏi với backend Java bạn sắp giỏi.

🧰 Chuẩn bị trước

  • Máy đã cài JDK 17 (khớp java.version=17 của project). Khuyên dùng bản LTS như Temurin/Adoptium 17.
  • IDE cho Java: IntelliJ IDEA (Community là đủ) — trải nghiệm tốt nhất cho Spring Boot; hoặc VS Code + Extension Pack for Java nếu muốn giữ môi trường quen thuộc.
  • Không cần cài Maven riêng — project dùng Maven Wrapper (mvnw.cmd trên Windows / mvnw trên Linux/Mac), version 3.9.10 sẽ tự tải.
  • MariaDB (tương thích MySQL) chạy local, dùng driver org.mariadb.jdbc.Driver. Vì ddl-auto: none, cần chạy file BE/sql.sql thủ công để tạo bảng trước khi khởi động app.
  • File .env ở thư mục BE: copy từ BE/env-template/.env-template rồi điền DATASOURCE_URL/USERNAME/PASSWORD/SCHEMA và JWT_SECRET_KEY (bắt buộc, không có app sẽ fail khi kết nối DB).
  • Kiến thức nền sẵn có (bạn đã có): Angular 17 (component, DI, @Injectable/@Component), TypeScript (interface, generic, kiểu tĩnh), RxJS cơ bản, và cách gọi REST API. Đây chính là vốn liếng để học nhanh.
  • Tâm thế: chấp nhận rằng vài chỗ ban đầu sẽ 'rườm rà' hơn frontend (kiểu tĩnh, biên dịch, cấu trúc thư mục sâu). Đây là đặc trưng backend doanh nghiệp, không phải bạn làm sai.
  • Tùy chọn nâng cao (có thể bỏ qua khi mới bắt đầu, đã được TẮT sẵn trong project): Redis, Kafka, Eureka, FTP — tài liệu sẽ chỉ rõ khi nào cần bật.

📖 Cách dùng tài liệu này

  • Học tuần tự từ Module 0 đến 10 — mỗi module xây trên nền module trước. Đừng nhảy cóc, đặc biệt với 4 module nền tảng (0–3); nắm chắc Dependency Injection ở Module 3 thì các tầng sau (Controller/Service/Repository) sẽ tự nhiên hiểu.
  • Luôn mở sẵn project thật BE/salary-service bên cạnh khi đọc. Mỗi khái niệm lý thuyết hãy lập tức tìm file tương ứng trong codebase (ví dụ: học @Service thì mở services/OptionSetServiceImpl.java) để thấy nó dùng thật như thế nào.
  • Tận dụng các bảng đối chiếu Angular/TS ↔ Java/Spring trong từng module — khi gặp khái niệm mới, hỏi trước: 'Cái này ở Angular là gì?'. Đa số đều có cặp tương đương.
  • Gõ lại code bằng tay, đừng chỉ copy-paste. Với người mới Java, việc tự gõ giúp não quen cú pháp tĩnh (kiểu dữ liệu, dấu chấm phẩy, import) nhanh hơn nhiều so với đọc lướt.
  • Sau mỗi module, tự đặt mục tiêu 'sửa được 1 thứ trong code thật': thêm 1 field vào DTO, thêm 1 endpoint GET, viết 1 query repository... Học qua việc thay đổi code thật bám rất chắc.
  • Khi bí, dùng GitNexus (đã cài cho repo) để tra cứu: gitnexus_query tìm luồng thực thi, gitnexus_context xem callers/callees của một symbol — thay vì đoán mò trong codebase lớn.
  • Chú ý các 'GOTCHA' được đánh dấu trong tài liệu (ví dụ: ddl-auto: none nên phải chạy sql.sql thủ công; profile mặc định là dev; phải điền .env trước khi chạy) — đây là những chỗ người mới hay vấp khi chạy app lần đầu.

🚀 Mẹo học nhanh cho dev frontend

  • TypeScript là bạn của bạn: kiểu tĩnh ở Java rất giống strict mode của TS. Bạn vốn ghét any — Java buộc bạn khai kiểu rõ ràng ở mọi nơi, đây là điểm cộng chứ không phải rào cản.
  • DI ở Spring gần như y hệt Angular: @Injectable() → @Service/@Component/@Repository, và cách inject qua constructor là giống nhau. Spring có 'IoC container' đóng vai trò như Angular injector — bạn đã hiểu mô hình này rồi.
  • pom.xml ≈ package.json + angular.json gộp lại: khai báo dependency và cấu hình build. ./mvnw ≈ npm/ng — chạy build, test, run. Không cần cài Maven riêng nhờ Maven Wrapper.
  • Đừng hoảng với 'boilerplate' của Java: project dùng Lombok (@Data, @Builder, @Getter/@Setter) để tự sinh getter/setter/constructor — bạn sẽ ít phải gõ tay hơn bạn tưởng.
  • Generic Java List<UserDTO> đọc y như Array<UserDTO>/UserDTO[] trong TS. Optional<T> chính là cách Java diễn đạt 'có thể null' — tương tự T | undefined mà bạn quen xử lý.
  • Annotation (@RestController, @Entity, @GetMapping) chính là decorator (@Component, @Input) ở Angular — cùng ý tưởng 'gắn metadata lên class/method để framework xử lý'.
  • Bất đồng bộ khác biệt: backend Spring chủ yếu code đồng bộ (blocking), KHÔNG cần Observable/RxJS như Angular. Một request = một luồng xử lý tuần tự từ trên xuống — ban đầu sẽ thấy 'đơn giản đến lạ' so với reactive ở frontend.
  • Học SQL song song một chút: JPA/Spring Data che bớt SQL, nhưng hiểu SQL cơ bản (SELECT/JOIN/WHERE) sẽ giúp bạn đọc các @Query(nativeQuery=true) trong UserRepository và debug nhanh hơn rất nhiều.
Module 0

Cài đặt & chạy được app (chiến thắng đầu tiên)

⏱ 3–4 giờ

Mục tiêu duy nhất của module này: làm cho project salary-service trong thư mục BE/ KHỞI ĐỘNG ĐƯỢC trên máy bạn — giống lần đầu bạn chạy ng serve cho một app Angular. Bạn sẽ cài đúng JDK 17, sửa JAVA_HOME, dùng Maven Wrapper (mvnw) thay vì cài Maven (giống npx chạy CLI mà không cài global), nạp biến môi trường từ file env/.env.dev, rồi chạy mvnw spring-boot:run và xác nhận app đã bind port 8062. Chưa cần hiểu sâu code — chỉ cần thấy nó chạy.

🎯 Mục tiêu

  • Cài JDK 17 (Temurin/Adoptium) và set JAVA_HOME đúng trên Windows; chạy java -version ra bản 17.x
  • Giải thích được Maven là gì so với npm, và vì sao mvnw/mvnw.cmd cho phép build mà KHÔNG cần cài Maven riêng (giống npx)
  • Hiểu cơ chế nạp biến môi trường của project này: DotenvInitializer đọc file env/.env.dev (chỉ định trong application-dev.yml), và biết file env-template/.env-template là bản mẫu để điền DATASOURCE_*, JWT_SECRET_KEY...
  • Chạy được mvnw spring-boot:run, đọc log khởi động (banner SMV, dòng 'Tomcat started on port 8062', 'Application started successfully')
  • Gọi thử app bằng trình duyệt/Postman và GIẢI THÍCH ĐÚNG vì sao response trả về 401/403 (do SecurityFilter chặn mọi request không có header gateway) — đó là dấu hiệu app ĐANG SỐNG, không phải lỗi

Module 1

Java căn bản cho dev TypeScript

⏱ 3–4 giờ

Bạn đã rành Angular + TypeScript, nên Java sẽ quen hơn bạn nghĩ: cùng họ ngôn ngữ C-style ({}, ;, class, interface, generic <T>), cùng tư duy kiểu tĩnh. Module này so từng dòng cú pháp Java với TypeScript bạn đã viết, dùng các file .java THẬT trong BE/ (project salary-service, Java 17) như Application.java, OptionSetValueDTO.java, Employees.java, DataUtils.java để bạn đọc hiểu được code Java ngay mà chưa cần học framework. Khác biệt cốt lõi: Java biên dịch ra bytecode chạy trên JVM (không phải transpile ra JS), kiểu tĩnh là THẬT lúc runtime (TS thì kiểu bị xoá khi chạy), và có sự phân biệt primitive vs object mà TS không có.

🎯 Mục tiêu

  • Đọc hiểu một file .java thật: nhận ra package, import, public class, và public static void main(String[] args) là gì và tương ứng với cái gì trong TS
  • Phân biệt được kiểu primitive (int, boolean, double, char) với kiểu object (Integer, String, Long) — điều TS không có
  • Khai báo biến/hàm/method trong Java và chỉ ra điểm khác so với let/const, function, và method TS
  • Giải thích package + import của Java tương ứng thế nào với module + import của TS, và vì sao đường dẫn package phải khớp cấu trúc thư mục
  • Dùng String và các method của nó (.length(), .substring(), .split(), .equals()), ép kiểu (cast) và var đúng cách
  • Tự viết và (về mặt khái niệm) biên dịch một class Java nhỏ có main() để in ra console

OOP & idiom Java (class, interface, generic, collection)

⏱ 3–4 giờ

⏱ 3–4 giờ

🎯 Mục tiêu

  • Giải thích được một HTTP request đi từ Angular HttpClient tới một method trong @RestController như thế nào, và đối chiếu với một Angular service đang gọi API đó.

  • Hiểu ResponseEntity<T> để chủ động chọn HTTP status, và vai trò của wrapper ExecutionResult<T> dùng chung trong dự án.
  • Viết DTO request có validation @NotBlank/@Size, kích hoạt bằng @Valid, và biết lỗi validation được biến thành response 400 ở GlobalExceptionHandler ra sao.
  • Tự thêm một endpoint mới (GET + POST) vào dự án salary-service chạy được trên nhánh practice.

@RequestMapping (base path)Giống prefix path bạn khai báo trong RouterModule.forRoot([{ path: 'order', ... }]) — đặt tiền tố chung cho mọi route con trong class đó.Đặt ở cấp class để gom các endpoint dưới một base path chung, ví dụ @RequestMapping("/order"). URL đầy đủ = context-path (/api/cab-service) + base path (/order) + path của method.
@GetMapping / @PostMapping / @PutMapping / @DeleteMappingTương ứng http.get(), http.post(), http.put(), http.delete() mà bạn gọi trong Angular service — nhưng ở phía nhận. Mỗi verb HTTP một annotation.Rút gọn của @RequestMapping(method = ...) cho từng HTTP method. @GetMapping("/v1/test") = method này chỉ đáp ứng HTTP GET tới path đó. Đây chính là verb mà Angular HttpClient phải gọi cho khớp.
@PathVariablePhần :id trong route Angular ({ path: 'order/:id' }) đọc qua ActivatedRoute.snapshot.params['id']. Ở client bạn nhúng nó vào URL: http.get(\/order/${id}\).Lấy một đoạn ĐỘNG nằm TRONG đường dẫn URL. Khai báo path "/{id}" rồi bind tham số @PathVariable Long id. Dùng cho định danh tài nguyên (id, code).
@RequestParamQuery param bạn gắn ở Angular qua HttpParams / { params: { page: 0, size: 20 } }, đọc lại bằng ActivatedRoute.queryParams.Lấy tham số từ phần query string sau dấu ? (vd ?optionSetCode=GENDER&page=0). Có thể đặt required=false và defaultValue. Hợp cho lọc/tìm kiếm/phân trang, không hợp làm định danh tài nguyên.
@RequestBodyChính là object bạn truyền làm body của http.post(url, body) trong Angular. Spring deserialize JSON body thành một DTO Java giống như TS nhận body: CreateOrderDto.Bind toàn bộ JSON body của request POST/PUT thành một object DTO Java (Jackson lo việc parse JSON). Dùng khi client gửi lên một payload có cấu trúc, vd form tạo mới.
ResponseEntity<T>Giống khi bạn không chỉ trả data mà còn quan tâm HttpResponse đầy đủ (status, headers) ở Angular qua { observe: 'response' }. Ở server bạn CHỦ ĐỘNG dựng response đó.Wrapper cho phép kiểm soát rõ HTTP status code, headers và body trả về. Vd ResponseEntity.ok(data) = 200, ResponseEntity.status(404).body(...). Khác với trả thẳng object (luôn 200).
DTO request/responseY hệt interface/class TS bạn khai báo cho payload và cho response model trong Angular (vd OrderCreateRequest, UserInfoResponse). DTO = hợp đồng dữ liệu giữa client và server.Plain object định nghĩa hình dạng dữ liệu đi vào (request) và đi ra (response). Tách khỏi @Entity (bảng DB) để không lộ field nhạy cảm. Trong dự án dùng Lombok @Data/@Builder để khỏi viết getter/setter.
@Valid + Bean Validation (@NotBlank, @Size)Giống Angular Reactive Forms Validators.required / Validators.maxLength(100) — nhưng chạy ở SERVER và bắt buộc, không thể bị bỏ qua như validate phía client.Đặt @Valid trước @RequestBody để Spring tự chạy các constraint khai báo trên DTO (@NotBlank, @Size(max=100)...). Nếu sai, Spring ném MethodArgumentNotValidException và GlobalExceptionHandler đổi thành response 400 có message i18n.
@RestControllerAdvice / @ExceptionHandlerGiống một Angular HttpInterceptor xử lý lỗi tập trung (catch error, map sang thông báo). Nhưng đây là phía server, bắt mọi exception từ controller.Class @RestControllerAdvice gom xử lý exception cho TẤT CẢ controller. Mỗi @ExceptionHandler(XxxException.class) map một loại lỗi sang một HTTP status + body chuẩn. Nhờ nó, controller không cần try/catch thủ công.

Module 5

Tầng dữ liệu — JPA & Spring Data

⏱ 3–4 giờ

Trong Angular, bạn gọi REST API qua HttpClient rồi map JSON về interface. Ở backend, tầng dữ liệu làm điều ngược lại: nó ánh xạ (map) các dòng trong bảng SQL thành object Java. JPA (qua Hibernate) là "ORM" lo việc map đó, còn Spring Data JPA cho bạn các Repository để CRUD và query mà gần như không phải viết SQL. Bài này bạn sẽ đọc các @Entity thật (Employees, OptionSet, OptionSetValue, User), đối chiếu với schema trong BE/sql.sql, hiểu JpaRepository, derived query (findByXxx) và @Query, rồi tự viết một truy vấn mới cho entity có thật.

🎯 Mục tiêu

  • Đọc hiểu một class @Entity và biết mỗi annotation (@Table, @Id, @GeneratedValue, @Column) ánh xạ ra cột/bảng nào trong BE/sql.sql
  • Phân biệt kiểu dữ liệu Java (Long, String, LocalDate, LocalDateTime, Integer, Boolean, BigDecimal) ↔ kiểu cột SQL (bigint, varchar, date, datetime, tinyint, decimal)
  • Giải thích @MappedSuperclass (BaseEntity) và lifecycle callback @PrePersist/@PreUpdate để tự set audit field
  • Hiểu vì sao project này KHÔNG dùng @OneToMany/@ManyToOne mà lưu khóa ngoại bằng ID (unitId, managerId, optionSetId) — và biết khi nào nên dùng quan hệ
  • Biết JpaRepository<Entity, ID> cho sẵn save/findById/findAll/delete/count mà không cần viết code
  • Tự viết được một derived query method (findByXxx) và một @Query (JPQL hoặc native) cho một entity có thật
  • Chạy được lệnh build Maven để xác nhận code mới biên dịch thành công

Module 6

Tầng Service & nghiệp vụ

⏱ 3–4 giờ

Tầng Service là nơi chứa "nghiệp vụ" (business logic) của backend — giống các @Injectable service trong Angular, nhưng chạy ở server và nói chuyện trực tiếp với database. Trong project salary-service, tầng này dùng pattern interface + impl: một interface trong services/iservice/ định nghĩa hợp đồng, một class ...Impl đánh dấu @Service hiện thực nó. Bạn sẽ học luồng Controller → Service → Repository, cách map Entity ↔ DTO, và ý nghĩa của @Transactional, rồi tự tay hiện thực một method nghiệp vụ thật trong OptionSetServiceImpl.

🎯 Mục tiêu

  • Hiểu vì sao backend tách riêng tầng Service, và nó tương ứng với @Injectable service nào trong Angular
  • Đọc và giải thích được pattern interface iservice + class Impl (@Service) trong codebase thật
  • Mô tả luồng dữ liệu Controller → Service → Repository → Database và chiều ngược lại Entity → DTO → JSON
  • Phân biệt được Entity (ánh xạ DB) với DTO (dữ liệu trả client), và biết khi nào dùng cái nào
  • Giải thích @Transactional làm gì và vì sao nó là thứ Angular service KHÔNG có
  • Tự hiện thực được một method nghiệp vụ nhỏ: lấy dữ liệu từ Repository, map sang DTO bằng Java Stream + Lombok @Builder

Module 7

Xử lý lỗi, i18n & chuẩn hoá response

⏱ 3–4 giờ

Module này dạy cách salary-service trả lỗi và response *thống nhất* cho toàn hệ thống — giống HttpInterceptor của Angular nhưng chạy ở phía server. Bạn sẽ học @RestControllerAdvice (bắt mọi exception ném ra từ controller/service và biến thành JSON chuẩn), custom exception (LogicException, AuthException, PermissionDeniedException), wrapper ExecutionResult<T> cùng ResponseEnricherAdvice (tự gắn timestamp/path vào mọi response), và cơ chế i18n đa ngôn ngữ (ResourceConfig + file lang_*.properties + LocaleInterceptor). Sau module, bạn hiểu trọn vòng đời: throw exception → handler bắt → tra message theo locale → bọc vào ExecutionResult → enrich → gửi về client.

🎯 Mục tiêu

  • Hiểu @RestControllerAdvice như một interceptor lỗi toàn cục: vì sao service chỉ cần throw exception mà không tự build JSON lỗi
  • Đọc hiểu 3 custom exception và biết khi nào ném loại nào (business 400, auth 401, permission 403)
  • Giải thích cơ chế i18n: từ keyMessage tới message dịch theo locale qua ResourceConfig và file lang_*.properties
  • Hiểu ExecutionResult<T> là 'envelope' chuẩn cho mọi response và cách ResponseEnricherAdvice bổ sung timestamp/path
  • Tự thêm được 1 exception handler mới, 1 message key i18n mới và test luồng lỗi trên codebase thật

Module 8

Bảo mật cơ bản với Spring Security

⏱ 3–4 giờ

Module này dạy cách Spring Security bảo vệ mọi request TRƯỚC KHI vào controller, qua một chuỗi filter (SecurityFilterChain). Hãy hình dung giống như HttpInterceptor + CanActivate guard của Angular, nhưng chạy ở phía server: mỗi request đi qua JWTFilter để xác định "user là ai" rồi mới được phép tới handler. Điểm đặc biệt của project salary-service: nó KHÔNG tự verify JWT mà TIN TƯỞNG các header (x-user-name, x-user-roles) do API Gateway đặt vào — đây là kiến trúc Gateway Pattern. Class JWTProvider có sẵn để ký/verify token nhưng hiện chưa được dùng trong luồng.

🎯 Mục tiêu

  • Giải thích được filter chain là gì và vì sao mọi request phải đi qua nó trước controller (liên hệ HttpInterceptor + guard ở Angular)
  • Đọc hiểu file cấu hình SecurityFilter.java: bật/tắt CORS, CSRF, stateless session, whitelist URL, và chèn filter custom vào chuỗi
  • Lần theo luồng xác thực thật: JWTFilter đọc header từ gateway → tạo UsernamePasswordAuthenticationToken → đặt vào SecurityContextHolder
  • Phân biệt Authentication (user là ai) và Authorization/phân quyền (user được làm gì) qua 3 role SUPER_ADMIN/ADMIN/STAFF
  • Hiểu khi auth thất bại thì AuthHandler trả về JSON 401/403 như thế nào
  • Nhận ra vì sao JWTProvider có code ký/verify JWT thật nhưng KHÔNG tham gia luồng hiện tại

  • 5. Tập trên nhánh practice, thao tác trên data test. UPDATE/DELETE luôn kèm WHERE; cân nhắc backup trước khi xoá.

3️⃣ Viết code — 7 lớp theo tầng (từ trong ra ngoài)

1Entity — ánh xạ bảng entities/
🎯 Mỗi cột = 1 field + 1 annotation @Column. ↔ giống khai một interface dữ liệu, nhưng có chú thích để JPA biết map vào đâu.
@Entity
@Table(name = "vocabulary")          // tên bảng trong DB
@Getter @Setter @Builder
@NoArgsConstructor @AllArgsConstructor
public class Vocabulary {

    @Id                                       // khóa chính
    @GeneratedValue(strategy = IDENTITY)      // ↔ AUTO_INCREMENT
    @Column(name = "vocabulary_id")
    private Long vocabularyId;

    @Column(name = "word",    length = 255) private String word;
    @Column(name = "meaning", length = 512) private String meaning;
    @Column(name = "part_of_speech")       private String partOfSpeech;  // snake_case ↔ camelCase
    // ... pronunciation, example, `level`, status, create/update datetime + user
}
2Repository — CRUD xuống DB repositories/
🎯 Chỉ cần extends JpaRepository là CÓ SẴN save / findById / findAll / deleteById — không viết thân hàm. ↔ như có sẵn 1 service gọi API mà không phải viết.
@Repository
public interface VocabularyRepository
        extends JpaRepository<Vocabulary, Long> {   // <Entity, kiểu khóa chính>

    // Spring tự sinh SQL TỪ TÊN method:
    //   WHERE LOWER(word) LIKE %?% OR LOWER(meaning) LIKE %?%
    List<Vocabulary> findByWordContainingIgnoreCaseOrMeaningContainingIgnoreCase(
            String word, String meaning);
}

4DTO — hình dạng vào/ra dto/
🎯 KHÔNG nhận/trả Entity trực tiếp. Request có ràng buộc @Valid; Response giấu bớt field nội bộ. ↔ tách interface body/response khỏi model nội bộ.
// dto/requests/VocabularyCreateRequest.java
@Data @Builder
public class VocabularyCreateRequest {
    @NotBlank(message = ResponseMessage.Vocabulary.MISSING_WORD)   // bắt buộc
    @Size(max = 255, message = ...INVALID_WORD_LENGTH)
    String word;
    @NotBlank(...) String meaning;
    @Size(max = 50, ...) String partOfSpeech;   // optional
}

// dto/VocabularyResponse.java
@Data @Builder
@JsonInclude(NON_NULL)            // field null thì bỏ khỏi JSON
public class VocabularyResponse {
    Long vocabularyId; String word; String meaning; String partOfSpeech;
    String level; String status; LocalDateTime createDatetime;
    // KHÔNG có create_user/update_user -> không lộ ra ngoài
}
5Service impl — logic + map services/
🎯 Nơi viết nghiệp vụ: build entity, save(), map sang DTO, không thấy thì throw LogicException (handler trả 400 i18n). ↔ thân của @Injectable service.
@Service
@RequiredArgsConstructor                       // ↔ constructor injection
public class VocabularyServiceImpl implements VocabularyService {

    private final VocabularyRepository repo;     // được inject vào

    @Transactional
    public VocabularyResponse create(VocabularyCreateRequest req) {
        Vocabulary v = Vocabulary.builder()
            .word(req.getWord()).meaning(req.getMeaning())
            // ...map các field...
            .status("1").createDatetime(LocalDateTime.now())
            .build();
        return toResponse(repo.save(v));            // INSERT rồi trả DTO
    }

    public VocabularyResponse getById(Long id) {
        Vocabulary v = repo.findById(id)
            .orElseThrow(() -> new LogicException(    // không thấy -> 400 + message i18n
                ResponseMessage.Vocabulary.NOT_FOUND));
        return toResponse(v);
    }
    private VocabularyResponse toResponse(Vocabulary v) { /* Entity -> DTO */ }
}
6Controller — định nghĩa URL controllers/
🎯 Map HTTP method ↔ hàm; bọc kết quả trong ExecutionResult. timestamp/path được chèn tự động. ↔ @PathVariable=route param, @RequestParam=query, @RequestBody=body POST.
@RestController
@RequestMapping("/vocabulary")                 // base path
@RequiredArgsConstructor
public class VocabularyController {

    private final VocabularyService service;

    @GetMapping                                 // GET /vocabulary?keyword=
    public ResponseEntity<ExecutionResult<List<VocabularyResponse>>> getAll(
            @RequestParam(required = false) String keyword) {
        return ResponseEntity.ok(ExecutionResult.<List<VocabularyResponse>>builder()
            .responseCode(Constants.ExecutionCode.SUCCESS)        // "0"
            .data(service.getAll(keyword)).build());
    }

    @PostMapping                                // POST /vocabulary
    public ResponseEntity<...> create(@Valid @RequestBody VocabularyCreateRequest req) {
        return ResponseEntity.ok(ExecutionResult...data(service.create(req)).build());
    }
    // @GetMapping("/{id}")  @PutMapping("/{id}")  @DeleteMapping("/{id}")
}

2Service đọc DB qua Repository 🔎 QUERY DB ← CHỖ QUERY DB

private Vocabulary findOrThrow(Long id) {

        .orElseThrow(() -> new LogicException(ResponseMessage.Vocabulary.NOT_FOUND));
}

🧠 Chốt 1 câu để nhớ mãi

  • Request đi XUỐNG theo tầng: Controller (nhận) → Service (nghĩ) → Repository (nói chuyện với DB) → MariaDB.
  • Dữ liệu đi LÊN theo tầng: Entity (sát DB) → DTO (sạch để trả ra) → ExecutionResult (vỏ bọc chuẩn) → JSON.
  • 2 lớp gác bao ngoài MỌI response: ResponseEnricherAdvice (luôn chèn timestamp/path) và GlobalExceptionHandler (mọi exception → JSON lỗi). Nhờ chúng, Controller/Service viết rất gọn.
  • Quy tắc dự án: đọc = GET, ghi (create/update/delete) = POST. Vòng đời như nhau, chỉ khác khúc giữa là SELECT hay INSERT/UPDATE/DELETE.
